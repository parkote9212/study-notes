### 2. Technical Deep Dive: 개념 및 동작 원리

### 2.1 IoC (Inversion of Control)

- **제어의 주도권 변경:** 과거에는 개발자가 직접 `new MemberService()`를 호출하여 객체를 생성했으나, 스프링에서는 컨테이너가 객체를 생성하고 관리합니다.
- **스프링 컨테이너:** `ApplicationContext`가 빈(Bean)을 생성하고 조립하는 역할을 수행합니다.

### 2.2 DI (Dependency Injection)

- **의존관계 주입:** 실행 시점(Runtime)에 외부에서 실제 구현 객체를 연결해 주는 것입니다.
- **주입 방식:** 생성자 주입(Constructor Injection)을 가장 권장하며, 이는 불변성(Immutability)과 테스트 용이성을 보장하기 때문입니다.

### 3. Critical Thinking: 기술적 의사결정 (A vs B)

| **비교 대상** | **수동 빈 등록 (@Bean)** | **자동 빈 등록 (@Component)** |
| --- | --- | --- |
| **설정 위치** | `@Configuration` 클래스 내부 | 각 클래스 헤더에 선언 |
| **제어권** | 기술적 설정이나 공통 로그 등 **외부 라이브러리** 빈 등록 시 유리 | 일반적인 비즈니스 로직(Service, Repository) 개발 시 유리 |
| **장점** | 빈 등록 과정을 한눈에 파악 가능 | 코드량이 획기적으로 줄어들어 생산성 향상 |

> "대부분의 비즈니스 로직은 **컴포넌트 스캔(@Component)**을 통해 자동 등록하되, 기술적인 공통 관심사나 외부 라이브러리 연동 시에만 **수동 빈 등록(@Bean)**을 사용하여 명확성을 확보합니다.”
>

### 4. Project Case Study: 실무 적용 사례 (STAR)

**🏀 스포츠센터 예약 시스템: 서비스-리포지토리 결합도 낮추기**

- **S (Situation):** 예약 로직(`ReservationService`)이 특정 데이터베이스 기술(예: `MemoryMemberRepository`)에 의존하고 있어 DB 교체가 어려운 상황.
- **T (Task):** 서비스 코드를 수정하지 않고도 저장소 기술을 교체할 수 있는 유연한 설계 필요.
- **A (Action):** `MemberRepository` 인터페이스를 설계하고, 서비스에서 인터페이스만 의존하도록 **생성자 주입(DI)** 적용.
- **R (Result):** 저장소 기술을 `MyBatis`에서 `JPA`로 변경하더라도 서비스 코드는 단 한 줄도 수정하지 않는 **OCP(개방-폐쇄 원칙)** 달성.

### 5. Interview Readiness: 면접 대비 및 회고

- **Q: IoC와 DI의 관계를 한 문장으로 설명한다면?**
    - A: IoC는 프로그램의 제어권이 넘어간 상태를 의미하는 추상적인 설계 원칙이며, DI는 그 원칙을 실제로 구현하기 위해 의존 객체를 주입해 주는 구체적인 기술입니다.
- **Q: DIP(의존관계 역전 원칙)를 스프링 없이 구현할 수 있나요?**
    - A: 네, 순수 자바 코드로도 인터페이스를 통해 구현할 수 있으나 스프링을 사용하면 컨테이너가 이 과정을 자동화해 주어 대규모 프로젝트에서 훨씬 안전하고 편리하게 관리할 수 있습니다.

### 🧐 회고 (Retrospective)

"왜 스프링을 쓰는가?"라는 근본적인 질문에 대한 답을 찾은 것 같습니다. 단순히 라이브러리 사용법을 아는 것보다, **객체 지향적인 설계**가 무엇인지 이해하는 것이 백엔드 개발자의 진짜 실력임을 깨달았습니다. 특히 생성자 주입 방식을 통해 불변성을 확보하는 습관을 프로젝트 전반에 적용해야겠습니다.
